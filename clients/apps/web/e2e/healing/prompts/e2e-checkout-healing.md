# E2E Flakiness Analysis

You are analyzing a failed E2E test to determine if it's a flaky test, a real regression, or an environment issue. You have access to Chrome DevTools MCP to reproduce the test steps in a real browser.

- **Branch:** ${BRANCH_NAME}
- **Trigger:** ${TRIGGER_EVENT}
- **Run URL:** ${RUN_URL}

## Instructions

### Step 1: Read the failing test

Use the `Read` tool to read the failing test spec file(s) in `clients/apps/web/e2e/` and any Page Object Model files in `clients/apps/web/e2e/pages/`. Understand what the test does step by step.

### Step 2: Reproduce the test manually

Using Chrome DevTools MCP, follow the exact same steps as the E2E test **at least 2 times** to determine consistency. The checkout URL is `${ENVIRONMENT_URL}/v1/checkout-links/${CHECKOUT_LINK}/redirect`. Read the test spec and page object model from Step 1 and replicate each action (navigation, clicks, fills, assertions) using the corresponding MCP tools (`navigate_page`, `click`, `fill`, `wait_for`, `take_snapshot`, `take_screenshot`, etc.).

After **every** action (navigation, click, fill, etc.), call `take_screenshot` and save the result to `healing-screenshots/` with a sequential filename like `01-navigate.png`, `02-select-country.png`, etc. This creates a visual log of the entire reproduction. Also use `take_snapshot` to verify the DOM state.

**Note:** Cross-origin iframes (e.g. Stripe Elements) cannot be interacted with via Chrome DevTools MCP. If the test fails inside an iframe, note this but don't try to interact with the iframe content.

### Step 3: Classify the failure

Based on your reproduction attempts, classify as one of:

- **flaky**: The test fails intermittently due to timing issues, race conditions, or brittle selectors. You observed the steps succeed at least once during reproduction.
- **regression**: The application behavior has genuinely changed. The test steps consistently fail in a way that indicates a real bug.
- **environment**: The failure is due to the test environment being unavailable, slow, or misconfigured. For example: API errors, timeouts on page load, missing test data.

### Step 4: Take action based on classification

#### If FLAKY:

1. Identify the root cause (timing, selector, race condition)
2. Create a fix by editing the test files. Common fixes include:
   - Adding `waitFor` or increasing timeouts for elements that load asynchronously
   - Using more resilient selectors (role-based over CSS class-based)
   - Adding retry logic for network-dependent assertions
   - Fixing race conditions by waiting for intermediate states
3. **Only modify files within `clients/apps/web/e2e/`** - never modify application source code
4. Create a branch and PR:

```bash
# Generate a slug from the test name
TEST_SLUG=$(echo "${FAILED_TESTS}" | head -1 | sed 's/[^a-zA-Z0-9]/-/g' | tr '[:upper:]' '[:lower:]' | head -c 40)
DATE=$(date +%Y%m%d)
BRANCH="fix/e2e-flaky-${TEST_SLUG}-${DATE}"

git checkout -b "$BRANCH"
git add clients/apps/web/e2e/
git commit -m "fix(e2e): fix flaky test - ${FAILED_TESTS}

Root cause: <describe the flakiness root cause>
Fix: <describe what was changed>

Co-Authored-By: Claude Code <noreply@anthropic.com>"

git push origin "$BRANCH"

gh pr create \
  --title "fix(e2e): fix flaky test" \
  --body "## Flaky Test Fix

**Test:** ${FAILED_TESTS}
**Classification:** Flaky
**Root cause:** <describe>
**Fix:** <describe>

**Reproduction results:**
- Attempt 1: <pass/fail + notes>
- Attempt 2: <pass/fail + notes>

**Triggered by:** ${RUN_URL}

---
Auto-generated by E2E Flakiness Analysis"
```
